# Architecture Smell Report

**Project:** CarrotGuard  
**Scope:** Entire cloned repository  
**Date:** 2026-06-28  
**Analyzer:** smell skill

---

## Executive Summary

CarrotGuard is a small Unity 2D tower-defense prototype organized into a mostly clean split between pure logic (`Core`, `Enemy`, `Tower`) and Unity adapters (`MonoBehaviour` classes plus `UI`). The pure C# classes are short, readable, and close to the PRD's phase structure.

The main risk is not low-level C# style; it is delivery and runtime integration. The repository contains scripts, tests, docs, and `.asmdef` files, but no checked-in Unity project settings, package manifest, scene, prefabs, or serialized assets. A fresh clone is therefore not a reproducible runnable prototype without manual reconstruction.

Tests cover many pure logic rules, but they do not cover Unity component lifecycle, prefab/scene wiring, UI interaction, or runtime ordering. Several production paths can fail or silently stop updating when Unity execution order or Inspector bindings differ from the README assumptions.

---

## Architectural Style Detected

Small layered Unity prototype:

| Expectation | Reality | Status |
|-------------|---------|--------|
| Pure gameplay rules isolated from Unity | Many core systems are pure C# and testable | Good |
| Unity adapters are thin wrappers | Some adapters stay thin, but `BuildMenu` and `Tower` contain runtime lookup and object creation details | Warning |
| Reproducible Unity project from clone | Missing `ProjectSettings`, `Packages`, `.unity` scene, prefabs, and serialized assets | Critical |
| Tests cover final playable behavior | Tests cover pure classes only; no PlayMode/runtime integration coverage | Warning |

## Findings by Category

### Critical / High

1. The repository is not a reproducible runnable Unity project from a fresh clone.
2. `Hud` can miss `LevelManager` subscription depending on Unity execution order.
3. Enemy path setup has unchecked null/length assumptions that can crash at runtime.

### Warnings

1. Tower target acquisition allocates and globally searches every frame.
2. Economy state is mutated through an exposed `GoldSystem` plus an `AwardGold(0)` refresh hack.
3. Tests rely heavily on reflection and miss Unity integration behavior.
4. Build/tower selection depends on undocumented collider, layer, and prefab Inspector wiring.

### Suggestions

1. Add duplicate/lifecycle protection for `LevelManager.Instance`.
2. Make `TowerUpgrade.SellRefund` idempotent or return a result object.
3. Convert established tests from reflection to direct strongly typed calls after the Red phase is complete.

## Detailed Findings

### Finding #1: Fresh Clone Is Not A Runnable Unity Prototype

- **Category:** Delivery / Architecture
- **Severity:** Critical
- **Location:** repository root; `README.md`
- **Evidence:** `rg --files -g "*.unity" -g "*.prefab" -g "ProjectSettings/**" -g "Packages/**" -g "*.asset"` found no scene, prefab, package manifest, project settings, or serialized assets.
- **Description:** The PRD's final deliverable is a Unity prototype that can run in the Editor. The repo currently requires the user to manually create the scene, prefabs, Canvas, EventSystem, path points, and serialized references following `README.md`. That makes the final behavior non-reproducible and prevents reliable test/manual verification from clone.
- **Recommendation:** Commit a minimal Unity project skeleton: `ProjectSettings/`, `Packages/manifest.json`, one sample `.unity` scene, enemy/tower/bullet prefabs, required colliders, Canvas, EventSystem, and serialized references. If this repo is intentionally code-only, say that explicitly and downgrade the delivery claim.

### Finding #2: HUD Subscription Can Be Lost

- **Category:** Unity Lifecycle
- **Severity:** Critical
- **Location:** `Assets/Scripts/UI/Hud.cs:13-39`, `Assets/Scripts/Core/LevelManager.cs:54-57`
- **Description:** `Hud.OnEnable()` calls `Subscribe()` once and returns when `LevelManager.Instance` is null. Unity does not guarantee cross-object `Awake` / `OnEnable` order unless script execution order is configured. If `Hud.OnEnable()` runs before `LevelManager.Awake()` assigns `Instance`, the HUD never subscribes and future gold/health changes do not refresh it.
- **Recommendation:** Subscribe in `Start()` after all `Awake()` calls, retry until `LevelManager.Instance` exists, or wire `LevelManager` explicitly. Also guard against duplicate subscriptions.

### Finding #3: Enemy Path Configuration Can Crash

- **Category:** Runtime Robustness
- **Severity:** Critical
- **Location:** `Assets/Scripts/Enemy/EnemySpawner.cs:31-45`, `Assets/Scripts/Enemy/Enemy.cs:64-73`, `Assets/Scripts/Enemy/PathFollower.cs:18-24`, `Assets/Scripts/Enemy/PathFollower.cs:42`
- **Description:** `EnemySpawner.SpawnEnemy()` checks `waypoints.Length == 0` but not `waypoints == null`, and it allows a one-point path. `Enemy.CreateWaypoints()` dereferences `waypoints.Length`, while `PathFollower` assumes `waypoints[0]` exists and later reads `waypoints[1]`.
- **Recommendation:** Validate path input once before spawning: non-null, length at least 2, and no null `Transform`s. Fail fast in `Awake` / `OnValidate` with a clear error and disable spawning if invalid.

### Finding #4: Tower Targeting Does Global Search And Allocation Every Frame

- **Category:** Performance / Complexity
- **Severity:** Warning
- **Location:** `Assets/Scripts/Tower/Tower.cs:34-67`
- **Description:** Every tower calls `FindObjectsOfType<Enemy>()` each `Update`, allocates an enemy array, allocates a `TargetPoint[]`, and creates one `TargetPoint` per enemy. Runtime complexity is roughly `O(towers * enemies)` every frame plus GC pressure. This is risky for mobile even in a small tower-defense prototype.
- **Recommendation:** Maintain an active enemy registry in `LevelManager` or a dedicated tracker updated from spawn/death/end events. Let towers iterate a shared list without `FindObjectsOfType`, and avoid per-frame wrapper allocations.

### Finding #5: Economy State Leaks Through `GoldSystem`

- **Category:** Design / Coupling
- **Severity:** Warning
- **Location:** `Assets/Scripts/Core/LevelManager.cs:42-45`, `Assets/Scripts/Core/LevelManager.cs:101-105`, `Assets/Scripts/UI/BuildMenu.cs:67-73`
- **Description:** `BuildMenu` upgrades by passing `LevelManager.Instance.GoldSystem` into `TowerUpgrade`, then calls `AwardGold(0)` to rebuild `BuildService` and raise `GoldChanged`. `AwardGold` replaces the entire `GoldSystem` object. This creates hidden stale-reference risk and makes a UI class responsible for knowing how to refresh economy state.
- **Recommendation:** Keep economy mutations inside `LevelManager`: add `TryUpgradeTower`, `SellTower`, and `AddGold` methods, or add `GoldSystem.Add`. Raise `GoldChanged` from the owner without replacing the `GoldSystem` object.

### Finding #6: Tests Do Not Cover Runtime Integration

- **Category:** Testing
- **Severity:** Warning
- **Location:** `Assets/Tests/EditMode/*.cs`
- **Evidence:** Tests repeatedly use `Type.GetType`, `Activator.CreateInstance`, and `GetMethod().Invoke`.
- **Description:** Reflection-based tests were useful during Red phases before classes existed, but at the final phase they avoid compile-time checking of public contracts and are hard to read. More importantly, there are no tests for `LevelManager`, `EnemySpawner`, `Enemy`, `Tower`, `Bullet`, `Hud`, `BuildMenu`, or `GameResultView` runtime interactions.
- **Recommendation:** Keep a few reflection tests only where they serve a specific compatibility purpose. Convert stable pure APIs to strongly typed tests, then add PlayMode or component tests for subscription order, spawn/register/death, tower build/upgrade/sell, HUD refresh, and victory/game-over display.

### Finding #7: Tower Selection Depends On Fragile Scene Wiring

- **Category:** UI / Runtime Robustness
- **Severity:** Warning
- **Location:** `Assets/Scripts/UI/BuildMenu.cs:168-171`, `Assets/Scripts/UI/BuildMenu.cs:94-111`, `Assets/Scripts/Tower/Tower.cs:10`, `README.md`
- **Description:** Existing tower selection uses `Physics2D.OverlapPoint()` and returns the first collider's `Tower` component. This requires every tower prefab to have a suitable `Collider2D`, assumes no other collider is hit first, and has no layer mask. Building also charges according to the selected button's `TowerType`, while the instantiated prefab's actual behavior comes from its serialized `towerType`.
- **Recommendation:** Document and enforce required colliders/layers in prefabs, or use a dedicated tower layer mask and `OverlapPointAll`. Consider configuring the spawned tower from the selected `TowerData` instead of relying only on Inspector state.

### Finding #8: `TowerUpgrade.SellRefund` Is Not Idempotent

- **Category:** Code / Domain Rule
- **Severity:** Suggestion
- **Location:** `Assets/Scripts/Tower/TowerUpgrade.cs:19-22`
- **Description:** `SellRefund` marks the tower as sold but does not check `IsSold` first. Calling it twice returns a refund twice. The current UI destroys the selected tower after selling, but the domain method itself does not protect the invariant.
- **Recommendation:** Return zero or a failure result for already sold towers, and add a unit test for double sell.

## Dependency Graph Analysis

The dependency direction is mostly acceptable for a small prototype:

- `Core` owns `LevelManager` and pure systems, but `LevelManager` imports `Enemy` and `Tower` to orchestrate runtime behavior.
- `Enemy` depends on `Core` only for `WaveSpawner` integration.
- `Tower` depends on `Enemy` for live target and damage behavior.
- `UI` depends on `Core` and `Tower`.

No obvious circular compile-time dependency was found in the source layout. The stronger issue is runtime global access through `LevelManager.Instance` and Unity scene lookup APIs.

## Refactoring Roadmap

### Immediate

1. Add a minimal runnable Unity project/scene/prefab set or clearly mark this repo as code-only.
2. Fix `Hud` subscription timing and add a lifecycle test.
3. Validate enemy waypoints before spawning and before constructing `PathFollower`.

### Short Term

1. Move build/upgrade/sell transactions into `LevelManager` or a small economy facade.
2. Replace `Tower.FindObjectsOfType` targeting with an active enemy registry.
3. Add PlayMode tests or scene smoke tests for start, spawn, build, kill, victory, game-over, restart.

### Later

1. Split `BuildMenu` into input selection, build command, and panel view responsibilities if UI grows.
2. Convert mature reflection tests to direct typed tests.
3. Add lightweight CI instructions for Unity batchmode test execution once a full project exists.
